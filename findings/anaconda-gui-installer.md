# Anaconda GUI Installer Feasibility

**Last updated:** 2026-07-22
**Status:** Research complete, not started

The installer ISO currently ships a text-only Anaconda experience. It asks for a username and password, wipes the disk, and that is it. An Azure Linux-themed graphical Anaconda installer would be a better experience - this doc captures what it would actually take.

## tl;dr

Anaconda GUI is not in the AZL 4.0 repos, but it is available from the Fedora 43 repos we already mix in. It installs, and it works. The hard blocker is `fedora-logos` as a mandatory dependency - Fedora branding gets baked into the installer. The fix is a stub or replacement package. Total overhead above text-only Anaconda is roughly 500 MiB extra on disk, 317 MiB downloaded. Feasible, but not a quick fix.

## What the AZL repos actually ship

AZL 4.0 (`mcr.microsoft.com/azurelinux-beta/base/core:4.0`) only packages text-mode Anaconda. Confirmed with tdnf in a container:

```
No match for argument: anaconda-gui
```

Text-only `anaconda` resolves to 200 packages, 500 MiB installed. That is the entire current installer dependency tree.

The GUI subpackage was split off upstream and AZL simply never packaged it. AZL dropped their custom installer in 3.0 and moved to plain Anaconda text mode - the server and container focus of the base distro makes a graphical installer irrelevant to them.

## With Fedora 43 repos mixed in

Since the installer ISO already mixes in Fedora 43 repos for the desktop packages, `anaconda-gui 43.44-3.fc43` resolves cleanly. The full dep tree from a mixed AZL 4.0 + Fedora 43 container:

| Package | Source | Size |
|---|---|---|
| `anaconda-gui` | fedora43 | 3.7 MiB |
| `anaconda-core` | fedora43 | 13.4 MiB |
| `anaconda-tui` | fedora43 | 727.7 KiB |
| `anaconda-widgets` | fedora43 | 204.3 KiB |
| `fedora-logos` | fedora43 | 1.7 MiB |
| `libreport-anaconda` | fedora43 | 19.2 KiB |

**Total: 317 MiB download, 1 GiB installed** (above the base text-only Anaconda tree).

The transaction resolves without conflicts. No GDM, no full GNOME session required - Anaconda GUI uses its own Wayland compositor.

## The fedora-logos problem

`fedora-logos` is a hard dependency of `anaconda-gui`. You cannot install the GUI without it. It ships Fedora branding into the installer environment - Fedora wordmark, the infinity logo, backgrounds. Not ideal when the installer should say "Azure Linux Desktop."

There are two ways to handle this:

1. **Stub it.** Create an `azurelinux-desktop-logos` package that satisfies the `fedora-logos` `Provides:` and ships AZL branding instead. This is exactly what every Fedora downstream does (RHEL, AlmaLinux, Rocky). It requires an RPM but it is a small one.

2. **Accept it temporarily.** Ship `fedora-logos` in the installer environment only - it does not end up on the installed system. The installed desktop gets none of it. The only place the Fedora branding appears is inside the installer itself, before first boot. Not great but not the end of the world for a proof of concept.

Option 2 is the fast path. Option 1 is the right answer long-term, but it is a new RPM and requires a spec and a build pipeline.

## Product name

Anaconda displays a product name during installation. By default it reads from `/etc/os-release`. Azure Linux Desktop already sets `PRETTY_NAME` correctly in the image, so the installed system name should be right. The installer environment itself gets the product name from the `product.img` shipped with the ISO. That would need to say "Azure Linux Desktop" rather than "Fedora."

The `product.img` is generated by `lorax` during the ISO build. Overriding it means either patching the lorax templates or generating a custom `product.img` as a build step.

## What enabling graphical mode actually requires

To flip the installer from text to GUI:

1. Add `anaconda-gui` and its deps to the installer ISO package list (in `kiwi/azl-desktop-installer.kiwi`).
2. Remove any `inst.text` from the installer kernel cmdline (check `kiwi/grub_template.cfg` and the KIWI image definition).
3. Handle `fedora-logos` via stub or acceptance.
4. Optionally provide a custom `product.img` for correct branding.

No Wayland compositor needs to be added separately - Anaconda GUI ships its own. No display manager needed.

## Why AZL probably dropped their custom installer

Azure Linux 2.0 had a custom installer. By 3.0 it was gone, replaced by text Anaconda. The reason is almost certainly that maintaining a custom installer for a distro whose primary delivery vehicles are containers and cloud images is not worth the effort. The graphical installer path is only meaningful for bare-metal desktop installs - which is exactly what this project is doing.

There is no fundamental technical blocker in AZL preventing a graphical Anaconda installer. It is a packaging and branding problem.

## Recommended approach

Before committing to this, scope the dep tree more carefully in a container - make sure there are no surprise conflicts between the Fedora 43 anaconda packages and the AZL 4.0 base libraries already on the installer ISO. The 317 MiB overhead estimate above is from a clean container; the real installer ISO already has most of GNOME installed, so the incremental cost is probably much less.

The logical path:

1. Spin up a container with the full installer ISO package list + `anaconda-gui`, confirm the transaction resolves cleanly.
2. Accept `fedora-logos` for a first cut, replace with an AZL stub later.
3. Add `anaconda-gui` to `kiwi/azl-desktop-installer.kiwi`, remove `inst.text`, rebuild.
4. Validate graphical mode boots and completes an install.
5. Build the stub `fedora-logos` replacement RPM and swap it in.

Steps 1-4 could ship in a single build iteration. Step 5 is a separate effort.
